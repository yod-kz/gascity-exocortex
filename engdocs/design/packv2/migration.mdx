This guide covers one focused PackV2 migration: moving each v1 `[[agent]]`
block out of `pack.toml` and into a v2 `agents/<name>/` directory with
`agent.toml` plus `prompt.template.md`.

For the broader city and pack migration path, see
[Shareable Packs](../../../docs/guides/shareable-packs.md).

## Why V2

In v1, an agent definition lived in a flat `[[agent]]` list while its prompt,
overlay, fragments, skills, and support files lived somewhere else. That made
agent review and per-agent changes harder than they needed to be: the identity
was in TOML, but the behavior was spread across pack-relative paths.

V2 makes the agent directory the unit of ownership. `agents/<name>/agent.toml`
holds config that differs from defaults. `agents/<name>/prompt.template.md`
holds templated prompt content, while `prompt.md` stays plain Markdown. Agent
local assets live beside those files, so a maintainer can version, copy, or
override one agent without scanning the whole pack.

## What Changes

| In v1 `pack.toml` | In v2 |
| --- | --- |
| `[[agent]] name = "worker"` | `agents/worker/` |
| `prompt_template = "prompts/worker.md"` | `agents/worker/prompt.template.md` for templated prompts |
| Scalar fields such as `scope`, `provider`, `wake_mode` | `agents/worker/agent.toml` |
| `append_fragments = ["footer"]` | `append_fragments` in `agent.toml`, with files in `template-fragments/` |
| `overlay_dir = "overlays/worker"` | `agents/worker/overlay/` or pack-wide `overlay/` |
| agent-local skills or MCP files | `agents/worker/skills/` and `agents/worker/mcp/` |
| shared prompt fragments | pack-wide `template-fragments/` |

## Migrate One Agent

Set the agent name once, then migrate one `[[agent]]` block at a time.

```bash
PACK_DIR=${PACK_DIR:-.}
AGENT=${AGENT:-worker}
cd "$PACK_DIR"
grep -n '^\[\[agent\]\]' pack.toml
```

```text
42:[[agent]]
```

Create the v2 directory and move the prompt into it. Use
`prompt.template.md` when the old prompt used Go template data or fragments;
use `prompt.md` only for plain Markdown.

```bash
mkdir -p "agents/${AGENT}"
PROMPT_SOURCE=${PROMPT_SOURCE:-prompts/${AGENT}.template.md}
cp -f "$PROMPT_SOURCE" "agents/${AGENT}/prompt.template.md"
```

Move scalar fields from the old `[[agent]]` block into `agent.toml`. Do not
copy `name`; the directory name is now the identity.

```toml
# agents/worker/agent.toml
scope = "rig"
provider = "codex"
wake_mode = "fresh"
append_fragments = ["operational-awareness"]
```

Move agent-local assets next to the agent. Leave shared assets at the pack
root.

```bash
mkdir -p "agents/${AGENT}/overlay" "agents/${AGENT}/template-fragments"
```

Delete the migrated `[[agent]]` block from `pack.toml`, then validate.

```bash
gc doctor
```

```text
No PackV1 agent layout collision warnings should be reported.
```

<Note>
Run `gc doctor` after each migrated agent. A one-agent loop keeps collisions
local and makes the final diff easier to review.
</Note>

<Warning>
Use `gc doctor --fix` only after reviewing or backing up the current pack. The
fix path is for safe mechanical rewrites, not for resolving semantic choices
such as which duplicate agent definition should win.
</Warning>

## Fragment-Authored Legacy Agents

`gc doctor --fix` handles root `city.toml` and root `pack.toml` legacy
`[[agent]]` tables because those rewrites have a clear destination:
`agents/<name>/agent.toml` in the same pack or city root.

Included TOML fragments are different. A fragment can be shared, layered, or
kept only for a specific deployment shape, so Gas City does not guess where a
fragment-authored legacy agent should move. When `gc doctor` reports a
fragment-authored legacy surface, migrate it by hand:

1. Create or choose the owning `agents/<name>/` directory in the pack or city
   that should own the agent.
2. Move scalar agent fields into `agents/<name>/agent.toml`.
3. Move prompt, overlay, namepool, skills, MCP, and template-fragment files
   beside that agent when they are agent-local.
4. Remove the legacy `[[agent]]` block from the fragment.
5. Run `gc doctor` again and keep the fragment only for non-agent deployment
   overrides that still belong there.

This is intentionally manual. If the right owner is ambiguous, make that
choice explicit in the repository rather than relying on an automatic rewrite.

## Auto-Imported System Packs

`internal/config/config.go` is the source of truth for generated Gastown city
imports. Today `GastownCity()` seeds only the `gastown` system pack, and both
entries point at `.gc/system/packs/gastown`.

| Generated surface | Binding | Source |
| --- | --- | --- |
| `[imports.gastown]` | `gastown` | `.gc/system/packs/gastown` |
| `[defaults.rig.imports.gastown]` | `gastown` | `.gc/system/packs/gastown` |

The first import composes city-scoped agents from the system pack. The default
rig import is applied when a new rig is added without explicit imports, so rig
agents come from the same canonical system pack path.

When `GastownCity()` changes this imports map, update this table in the same
change.

## `fallback = true`

`fallback = true` is a pack composition escape hatch for duplicate agent names.
The implementation is `resolveFallbackAgents` in `internal/config/pack.go`:

```go
func resolveFallbackAgents(agents []Agent) []Agent
```

Resolution is based on the agent's qualified name and source directory.
Duplicates from the same source directory are not handled here; they are
reported by the normal duplicate-agent validation path.

1. If at least one fallback and one non-fallback agent with the same qualified
   name come from different source directories, every fallback copy is removed.
2. If all colliding copies are fallbacks, the first loaded copy wins. Load order
   is depth-first include/import order.
3. If no copy is a fallback, the collision remains and validation reports an
   error.
4. A city-level inline agent has an empty source directory, which still counts
   as distinct from a system pack source. That lets a city-defined non-fallback
   agent shadow a system fallback.

```toml
# imported pack
[[agent]]
name = "dog"
fallback = true
```

```toml
# city pack
[[agent]]
name = "dog"
```

In that composition, the city pack's `dog` remains active and the imported
fallback `dog` is discarded before duplicate-agent validation runs.

## Worked Example: Gastown `witness`

Gastown's `witness` agent is already in the v2 shape. A v1 equivalent would
have put the identity and scalar fields in `pack.toml`:

```toml
# v1 pack.toml
[[agent]]
name = "witness"
scope = "rig"
prompt_template = "prompts/witness.template.md"
wake_mode = "fresh"
work_dir = ".gc/agents/{{.Rig}}/witness"
nudge = "Run 'gc prime' to check worker status and begin patrol cycle."
idle_timeout = "1h"
max_session_age = "5h"
max_session_age_jitter = "15m"
max_active_sessions = 1
```

The v2 pack removes that block and stores the agent under its own directory:

```text
examples/gastown/packs/gastown/agents/witness/
├── agent.toml
└── prompt.template.md
```

```toml
# agents/witness/agent.toml
scope = "rig"
wake_mode = "fresh"
work_dir = ".gc/agents/{{.Rig}}/witness"
nudge = "Run 'gc prime' to check worker status and begin patrol cycle."
idle_timeout = "1h"
max_session_age = "5h"
max_session_age_jitter = "15m"
max_active_sessions = 1
```

The prompt body moves into `agents/witness/prompt.template.md`. The `name`
field disappears because `witness` is now derived from the directory path.

## Verification

Run `gc doctor` and expect no PackV1 layout collision for the migrated agent.
Then inspect or start the city and confirm the prompt path resolves
from `agents/<name>/`.

```bash
gc doctor
gc config show
```

## Troubleshooting

### Agent Not Found

Check that `agents/<name>/agent.toml` or a prompt file exists. An empty
directory is not an agent definition.

```bash
find agents -maxdepth 2 -type f | sort
```

### Fallback Agent Still Appears

Check whether the local agent is also marked `fallback = true`. A non-fallback
agent shadows fallbacks; two fallback definitions keep the first loaded copy.

### `pack.toml` Decode Error

Run `gc doctor` for the exact file and line, then inspect the block you edited.
Most failures come from leaving a partial `[[agent]]` table behind.

### Auto-Imported Pack Missing

Confirm the generated import source still matches
`.gc/system/packs/gastown`. If `GastownCity()` changed, update the table above
and the generated config together.

### Fragments Do Not Render

Only `.template.md` files run through Go templates. If a prompt needs
`{{ template "fragment" . }}`, the prompt file must be named
`prompt.template.md`.

<Note>
See also: If you're hitting an error other than `duplicate name`, see the

</Note>
